## 2. 技术选型

### 2.1 AI 模型/框架

| 能力 | 选型方案 | 说明 |
|------|----------|------|
| **意图识别** | GPT-4o-mini / Qwen-Max | Few-shot 分类，支持自定义意图类别 |
| **实体抽取** | GPT-4o-mini + Function Calling | 结构化输出，抽取公司/行业/痛点等 |
| **主题聚类** | BGE-M3 + Qdrant 向量检索 | 语义相似度聚类 |
| **情感分析** | 预设规则 + LLM 判断 | 结合语气词、关键词、上下文 |
| **OCR 文档解析** | PaddleOCR + Marker PDF | 高精度文档结构化 |

### 2.2 后端框架

| 组件 | 选型 | 理由 |
|------|------|------|
| **Web 框架** | FastAPI | 高性能、自动 OpenAPI、异步支持 |
| **任务队列** | Celery + Redis | 成熟的分布式任务调度 |
| **API 网关** | Kong / Nginx | 限流、认证、日志 |
| **ORM** | SQLAlchemy 2.0 | 类型安全、异步支持 |

### 2.3 前端框架

| 组件 | 选型 | 理由 |
|------|------|------|
| **框架** | React 18 + TypeScript | 组件化、生态成熟 |

| **状态管理** | Zustand | 轻量、支持中间件 |

### 2.4 存储选型

| 用途 | 选型 | 说明 |
|------|------|------|
| **结构化数据** | PostgreSQL 16 | JSON 支持、事务完整 |
| **文件存储** | MinIO (S3 兼容) | 自部署、成本低 |
| **缓存** | Redis 7 | 会话缓存、任务队列后端 |
| **向量检索** | Qdrant | 高性能、Rust 实现 |

## 3. 数据流设计

### 3.1 完整数据流转图

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                              数据流转全链路                                     │
└──────────────────────────────────────────────────────────────────────────────┘

[数据导入] ──► [格式解析] ──► [隐私脱敏] ──► [数据清洗] ──► [AI 分析]
                                    │                    │
                              ┌─────┴─────┐        ┌─────┴─────┐
                              ▼           ▼        ▼           ▼
                        [原始数据存储]  [敏感字段]  [分析中间结果] [向量嵌入]
                              │                      │
                              └──────────┬───────────┘
                                         ▼
                              ┌─────────────────────┐
                              │     画像生成与更新     │
                              └──────────┬──────────┘
                                         ▼
                              ┌─────────────────────┐
                              │    画像版本快照       │
                              └──────────┬──────────┘
                                         ▼
                              ┌─────────────────────┐
                              │    画像展示输出      │
                              └─────────────────────┘
```

### 3.2 各环节详细说明

**阶段一：数据导入**
- 支持 JSON/TXT/CSV 聊天记录批量导入
- 支持 Word/PDF/Markdown 需求文档解析
- 文件上传至 MinIO，触发异步处理任务

**阶段二：格式解析与脱敏**
- 解析文件结构，提取 messages/content/role/timestamp
- 正则 + NLP 识别手机号、身份证、邮箱等敏感字段
- 替换为占位符（如 `138****1234`），保留数据格式

**阶段三：AI 分析**
- 意图识别：LLM Few-shot 分类
- 实体抽取：Function Calling 结构化输出
- 主题聚类：BGE-M3 向量化 + Qdrant 聚类
- 情感分析：规则 + LLM 上下文判断

**阶段四：画像生成**
- 按客户维度聚合分析结果
- 置信度计算（基于数据量、时间衰减）
- 标签权重排序，输出结构化画像

**阶段五：输出展示**
- 文字标签云 + 维度雷达图
- 版本历史对比
- 数据来源追溯

## 4. 核心算法说明

### 4.1 意图识别

**实现思路：**
1. 构建意图分类体系：咨询、投诉、合作、询价、投诉、闲聊
2. Few-shot Prompt 设计，提供 2-3 条示例
3. LLM 返回意图类别 + 置信度
4. 多轮对话时聚合时间窗口内意图分布

**Prompt 示例：**
```
你是一个客服对话意图分类器。请判断用户消息的意图类别。

可选类别：咨询、投诉、合作、询价、闲聊

示例：
用户：你们产品多少钱？
意图：询价

用户：产品质量有问题
意图：投诉

请判断：
用户：{user_message}
意图：
```

### 4.2 实体抽取

**实现思路：**
1. 定义实体 Schema：公司名称、行业、公司规模、预算、痛点、期望、时间节点
2. 使用 Function Calling / JSON Mode 结构化输出
3. 多轮抽取结果合并去重

**抽取 Schema：**
```json
{
  "company": {"type": "string", "description": "公司名称"},
  "industry": {"type": "string", "description": "所属行业"},
  "size": {"type": "string", "description": "公司规模，如100-500人"},
  "budget": {"type": "string", "description": "预算范围"},
  "pain_points": {"type": "array", "items": "string", "description": "痛点问题"},
  "expectations": {"type": "array", "items": "string", "description": "期望方案"}
}
```

### 4.3 情感分析

**实现思路：**
1. 规则层：关键词匹配（积极词/消极词）+ 感叹号/问号统计
2. LLM 层：上下文综合判断，输出 5 档情感评分
3. 聚合层：多轮对话加权平均

**情感档位：**
| 档位 | 分值 | 表现特征 |
|------|------|----------|
| 非常积极 | 5 | 明确合作意向、高度认可 |
| 积极 | 4 | 主动咨询、态度友好 |
| 中性 | 3 | 客观询问、无明显情绪 |
| 消极 | 2 | 抱怨、质疑 |
| 非常消极 | 1 | 强烈不满、拒绝 |

## 5. 存储方案

### 5.1 数据分层存储

| 数据类型 | 存储方案 | 生命周期 |
|----------|----------|----------|
| **原始数据** | MinIO 对象存储 | 永久保留 |
| **脱敏数据** | MinIO + PostgreSQL | 永久保留 |
| **分析结果** | PostgreSQL | 永久保留 |
| **向量嵌入** | Qdrant | 可定期重建 |
| **会话缓存** | Redis | 7 天过期 |
| **版本快照** | PostgreSQL JSONB | 永久保留 |

### 5.2 数据库 Schema 设计

**客户信息表 (customers)**
```sql
CREATE TABLE customers (
    id UUID PRIMARY KEY,
    name VARCHAR(255),          -- 脱敏后
    company VARCHAR(255),
    industry VARCHAR(100),
    size VARCHAR(50),
    source_type VARCHAR(20),    -- 'chat' / 'document'
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

**会话记录表 (conversations)**
```sql
CREATE TABLE conversations (
    id UUID PRIMARY KEY,
    customer_id UUID REFERENCES customers(id),
    conversation_id VARCHAR(100),  -- 原始ID
    date DATE,
    raw_data JSONB,                -- 原始JSON
    sanitized_data JSONB,         -- 脱敏后
    status VARCHAR(20),            -- 'pending'/'processing'/'done'
    created_at TIMESTAMP
);
```

**画像表 (profiles)**
```sql
CREATE TABLE profiles (
    id UUID PRIMARY KEY,
    customer_id UUID REFERENCES customers(id),
    version INT,
    attributes JSONB,              -- 画像维度数据
    confidence_scores JSONB,       -- 各维度置信度
    source_conversations UUID[],  -- 关联会话
    created_at TIMESTAMP
);
```

**画像版本历史表 (profile_versions)**
```sql
CREATE TABLE profile_versions (
    id UUID PRIMARY KEY,
    profile_id UUID REFERENCES profiles(id),
    version INT,
    snapshot JSONB,               -- 版本快照
    change_summary TEXT,          -- 变更说明
    created_at TIMESTAMP
);
```

### 5.3 文件存储结构

```
minio-bucket/
```

## 6. 安全与隐私设计

### 6.1 隐私脱敏流程

```
原始数据 ──► 敏感信息识别 ──► 脱敏替换 ──► 脱敏验证 ──► 存储
                │                  │
                ▼                  ▼
           [正则+NLP]        [手机/身份证/邮箱/姓名]
```

**脱敏规则：**
| 敏感类型 | 识别方式 | 脱敏规则 |
|----------|----------|----------|
| 手机号 | 正则 `\d{11}` | `138****1234` |
| 身份证 | 正则 `\d{17}[\dXx]` | `310***********1234` |
| 邮箱 | 正则 `[a-zA-Z0-9._%+-]+@[a-zA-Z0-9.-]+` | `zhang***@company.com` |
| 姓名 | NLP 命名实体识别 | `张总` → `客户A` |
| 地址 | NLP + 正则 | 模糊化处理 |

### 6.2 数据加密

| 层面 | 方案 |
|------|------|
| **传输加密** | HTTPS (TLS 1.3) |
| **存储加密** | MinIO 服务端加密 (SSE-S3) |
| **数据库加密** | PostgreSQL Transparent Data Encryption |
| **敏感字段** | 应用层 AES-256 加密 |

### 6.3 权限控制

**角色权限体系：**
| 角色 | 数据查看 | 数据导入 | 画像编辑 | 系统管理 |
|------|----------|----------|----------|----------|
| 管理员 | ✅ 全部 | ✅ | ✅ | ✅ |
| 产品经理 | ✅ 所在项目 | ✅ | ✅ | ❌ |
| 运营 | ✅ 所在项目 | ✅ | ❌ | ❌ |
| 访客 | ✅ 仅自己 | ❌ | ❌ | ❌ |

**实现方式：**
- JWT Token 认证
- RBAC 权限模型
- 数据行级权限（项目隔离）

### 6.4 合规保障

- 操作日志完整记录（谁、什么时间、做了什么）
- 数据删除支持级联删除（含向量数据）
- 定期安全审计

## 7. 迭代机制实现

### 7.1 增量更新流程

```
新数据导入
    │
    ▼
变更检测 ──► 对比已有画像数据
    │
    ├── 无变化 ──► 跳过
    │
    └── 有变化 ──► 差异分析
                      │
                      ▼
               AI 重新分析变更点
                      │
                      ▼
               画像维度更新
                      │
                      ▼
               版本快照记录
```

### 7.2 触发机制实现

**实时增量（事件驱动）：**
```python
# 文件上传触发
@router.post("/upload")
async def upload_file(file: UploadFile):
    task = await enqueue_analysis.delay(file_id)
    return {"task_id": task.id}

# Celery Task
@celery.task
def analysis_task(file_id):
    # 1. 读取文件
    # 2. 脱敏处理
    # 3. AI 分析
    # 4. 变更检测
    # 5. 更新画像
    # 6. 记录版本
```

**周期合并（定时任务）：**
```python
# Celery Beat 配置
CELERYBEAT_SCHEDULE = {
    "weekly-merge": {
        "task": "merge_profiles",
        "schedule": crontab(hour=2, minute=0, day_of_week=1),
    },
}
```

### 7.3 置信度计算

```
confidence = base_score * data_volume_factor * recency_factor * consistency_factor

- base_score: 0.5 (初始值)
- data_volume_factor: min(1.0, data_count / 10)  # 10条数据饱和
- recency_factor: 0.5 + 0.5 * exp(-0.1 * days_old)  # 指数衰减
- consistency_factor: 相同标签出现次数 / 总提及次数
```

## 9. 技术风险与应对

### 9.1 风险矩阵

| 风险 | 概率 | 影响 | 优先级 | 应对方案 |
|------|------|------|--------|----------|
| **LLM API 成本超预期** | 高 | 中 | P1 | Prompt 缓存、结果复用、GPT-4o-mini 优先 |
| **文档解析准确率不足** | 中 | 高 | P1 | 人工标注训练数据、PDF 解析选型 |
| **数据孤岛（客户无法关联）** | 中 | 高 | P2 | 唯一标识规范、模糊匹配辅助 |
| **冷启动置信度低** | 高 | 低 | P3 | 引导界面、低置信度提示文案 |
| **并发导入性能瓶颈** | 中 | 中 | P2 | 异步队列、限流策略 |
| **隐私泄露风险** | 低 | 极高 | P0 | 强制脱敏、审计日志、加密存储 |
| **向量数据库扩展性** | 低 | 中 | P3 | Qdrant 分片、PGVector 备选 |

### 9.2 详细应对措施

**LLM API 成本控制：**
1. 结果缓存（Redis，7 天有效期）
2. 批量处理减少 API 调用次数
3. 优先使用 GPT-4o-mini，复杂任务才用 GPT-4o
4. 设置月度预算告警

**文档解析准确率：**
1. PDF 解析使用 Marker PDF（高准确率）
2. Word 解析使用 python-docx
3. 预留人工标注接口，错误样本持续优化

**客户数据关联：**
1. 导入时强制填写客户唯一标识
2. 提供「合并客户」功能
3. 模糊匹配（公司名相似度）辅助识别

**冷启动策略：**
1. 新建客户时展示「数据不足」提示
2. 3 条以下数据标注「样本较少」
3. 引导用户补充更多数据源

### 9.3 监控指标

| 指标 | 告警阈值 |
|------|----------|
| LLM API 响应时间 | > 10s |
| 单次分析成本 | > ¥0.1 |
| 队列积压任务数 | > 100 |
| 脱敏失败率 | > 1% |
| 画像更新失败率 | > 0.1% |
