# Task Closure SOP — 任务闭环标准流程

## 触发条件
当任务满足以下任一条件时，强制执行本 SOP：
- 任务需要 3 步以上操作
- 任务有明确交付物（文件/代码/报告）
- 任务结果需要用户确认

## 七步闭环流程

### Step 1 — 目标确认
```
[ ] 用 clarify 确认：最终交付物是什么？
[ ] 确认成功标准：什么样算完成？
[ ] 确认截止时间（如果有）
```
**输出**：「任务目标 + 成功标准」确认记录

---

### Step 2 — 规划步骤
```
[ ] 把任务拆解成 N 个有序步骤（不超过7步）
[ ] 识别每步的检查点（checklist item）
[ ] 预估每步的工具和耗时
[ ] 如果步骤 > 7，拆分或委托子任务
```
**输出**：带编号的步骤清单

### Step 2.5 — 一个一个处理（用户偏好）
主人倾向于**一次只处理一个阻塞项**，不允许提前并行启动多个方向。
收到任务后：先列出所有阻塞项 → 问主人按什么顺序处理 → 按主人指定的顺序一个个解决，不要并行探索多个方向。
**例外**：明确说"可以同时做"的才能并行。

---

### Step 3 — 执行 + 检查点
```
每执行一步后：
[ ] 对照检查点确认该步完成
[ ] 记录关键发现（toolong → 写到临时文件）
[ ] 如果检查点失败 → 记录原因 → 重试或调整
```
**输出**：每步的「执行记录 + 检查点结果」

---

### Step 4 — 验证输出
```
[ ] 交付物存在且非空
[ ] 格式正确（文件类型、内容结构）
[ ] 自查：如果我是用户，会满意这个结果吗？
[ ] 如果不满意 → 回退到 Step 2 重新规划
```
**输出**：验证结论（通过 / 需返工）

---

### Step 5 — 异常处理（最多3次重试）
```
第1次失败：
  → 记录失败原因
  → 调整策略（换工具 / 换方法）
  → 重试

第2次失败：
  → 记录失败原因
  → 尝试完全不同路径
  → 重试

第3次失败：
  → 停止重试
  → 汇报：任务卡在哪里 + 已尝试的方法 + 建议方案
  → 升级给用户决策
```
**输出**：「异常记录 + 尝试路径 + 最终状态」

---

### Step 6 — 结果汇报
```
格式：
📋 任务状态：✅ 完成 / ⚠️ 部分完成 / ❌ 失败
📦 交付物：[文件路径或内容摘要]
📝 执行摘要：[N] 步，[X] 次重试，[Y] 个检查点通过
💡 备注：[如果有的话]
```
**输出**：结构化汇报

---

### Step 7 — 交接确认
```
[ ] 用户确认收到 / 满意
[ ] 如果需要后续行动 → 创建 cronjob 或记录到 MEMO
[ ] 更新当天日记（当天日记文件追加记录）
```
**输出**：交接确认记录

---

## 注意事项

- **复杂任务不要跳步**：特别是 Step 1（目标确认）和 Step 2（规划）看似花时间，实际省时间
- **步骤 > 7 时**：拆分任务，用 delegate_task 并行执行子任务
- **重试记录**：每次重试都简短记录原因（1句话），便于后续审计
- **toolong 文件**：如果某个步骤输出超过 50 行，写到 `~/.hermes/tmp/task_<timestamp>.txt`，汇报里只说「已保存到 X」
