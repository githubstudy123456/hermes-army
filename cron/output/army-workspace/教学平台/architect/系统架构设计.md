# 系统架构设计方案

## 1. 架构设计原则

### 1.1 核心原则
- **模块化设计**：各学科独立模块，便于单独开发和维护
- **可扩展性**：支持新增学科不影响现有系统
- **高性能**：支持大规模用户并发访问
- **前后端分离**：前端独立部署，后端API化

### 1.2 技术选型
- 前端：React/Vue + TypeScript
- 后端：Node.js + Express/NestJS
- 数据库：PostgreSQL + Redis
- 3D渲染：Three.js / Babylon.js
- 文件存储：OSS/S3兼容存储

## 3. 多学科扩展方案

### 3.1 学科模块设计
```
subjects/
├── physics/          # 物理学科
│   ├── mechanics/    # 力学
│   ├── thermodynamics/ # 热学
│   ├── optics/       # 光学
│   ├── electromagnetism/ # 电磁学
│   └── modern/       # 近代物理
├── chemistry/        # 化学学科
├── biology/         # 生物学科
└── mathematics/     # 数学学科
```

### 3.2 通用领域模型
```typescript
// 跨学科通用内容结构
interface SubjectContent {
  id: string;
  subject: SubjectType;
  chapter: string;
  section: string;
  knowledgePoints: KnowledgePoint[];
  videos: Video[];
  exercises: Exercise[];
  experiments: Experiment[];
}

// 学科特有扩展
interface PhysicsContent extends SubjectContent {
  formulas: Formula[];
  diagrams: Diagram[];
  models3d: Model3D[];
}
```

### 3.3 插件化架构
- 每个学科作为独立插件开发
- 统一的内容接口规范
- 共享的组件库和服务

### 4.4 3D服务 (3D Service)

- 渲染服务

## 5. 数据库设计

### 5.1 核心表结构
```sql
-- 学科表
CREATE TABLE subjects (
  id VARCHAR(36) PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  code VARCHAR(50) UNIQUE NOT NULL,
  icon VARCHAR(255),
  sort_order INT DEFAULT 0,
  created_at TIMESTAMP
);

-- 知识点表
CREATE TABLE knowledge_points (
  id VARCHAR(36) PRIMARY KEY,
  subject_id VARCHAR(36) REFERENCES subjects(id),
  chapter VARCHAR(100),
  section VARCHAR(100),
  title VARCHAR(200),
  content JSONB,
  difficulty INT DEFAULT 1
);

-- 视频课程表
CREATE TABLE video_courses (
  id VARCHAR(36) PRIMARY KEY,
  subject_id VARCHAR(36) REFERENCES subjects(id),
  knowledge_point_ids VARCHAR[],
  title VARCHAR(200),
  duration INT,
  url VARCHAR(500)
);
```

## 6. API设计

### 6.1 RESTful API结构
```
GET    /api/v1/subjects              # 获取学科列表
GET    /api/v1/subjects/:id          # 获取学科详情
GET    /api/v1/subjects/:id/content # 获取学科内容
POST   /api/v1/learning/progress     # 更新学习进度
GET    /api/v1/learning/report       # 获取学习报告
```

### 6.2 GraphQL可选方案
- 对于复杂查询场景可选用GraphQL
- 移动端优先场景适用
