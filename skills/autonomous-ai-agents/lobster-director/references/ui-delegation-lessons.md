# UI/Frontend 委派规范

## 核心原则：先确认，再执行

物理平台这类 UI 改版任务，必须走「确认 → 执行 → 验收」三步：

```
用户反馈UI问题
  ↓ 理解需求，在聊天里描述自己的理解
用户确认 ✓
  ↓ delegate_task 给 lobster-dev-frontend 执行
  ↓ 等待结果
  ↓ build + 重启服务
  ↓ 用户验收
```

**绝对不要**：自己揣摩用户意思，直接派任务让 lobster-dev 批量改完，再等用户反馈。这样会浪费两轮（改完发现方向不对再改回来）。

## 具体操作

当用户对 UI 提出问题时：
1. 用「这样对吗？」格式描述自己的理解
2. 等待用户一句话确认（或纠正）
3. 只有确认后才 delegate_task 给 lobster-dev

示例：
- ❌ 用户说"按钮没用"，直接派给 lobster-dev 改
- ✓ 用户说"按钮没用"，先问"是点击没反应还是样式问题？"，等回答后再派

## 为什么这条规则重要

lobster-dev-frontend 是 leaf agent，不会在执行前提问。它只接收 prompt 中的指令。如果 prompt 里的理解是错的，改完发现不对就得重来，浪费一次 build + 重启循环。

## 关联教训

- 「不要一口气吃成大胖子」→ 一次只改一个问题，等确认再改下一个
- 「你一个一个来」→ 同上，按顺序逐个处理，不要并行走多步
