# Breach Response · 数据泄露响应

## 基本信息
- **Skill Name**: breach-response
- **Description**: 数据泄露事件响应流程，支持通知制作和监管报告
- **Trigger**: 发现数据泄露 / 收到安全事件报告 / 监管调查启动
- **Category**: privacy-legal

---

## 核心能力

### 1. 中国PIPL数据泄露报告要求
| 要求 | 具体内容 |
|------|---------|
| 报告时限 | 发现后72小时内向网信部门报告 |
| 报告内容 | 泄露原因、泄露数据类型和数量、可能影响 |
| 个人通知 | 应当及时通知个人（无法通知时公告） |
| 保存记录 | 泄露事件记录保存备查 |
| 补救措施 | 立即采取补救措施 |

### 2. GDPR数据泄露报告要求（对比）
| 要求 | GDPR | PIPL |
|------|------|------|
| 报告时限 | 72小时 | 72小时 |
| 监管机构 | 主管 supervisory authority | 网信部门 |
| 个人通知 | 当发现可能高风险时 | 应当及时通知 |
| 记录保存 | 必须记录 | 应保存备查 |

### 3. 响应流程
```
发现事件 → 初步评估 → containment（遏制） → 影响评估 → 监管报告 → 个人通知 → 事后复盘
```

### 4. 泄露评估矩阵
| 影响等级 | 评估标准 | 响应级别 |
|---------|---------|---------|
| 高 | 敏感信息大量泄露、可能造成人身/财产损害 | 即时响应+高层批准 |
| 中 | 一般个人信息泄露、受影响人数有限 | 24小时内响应 |
| 低 | 内部日志/非敏感信息泄露 | 72小时内处理 |

---

## 操作步骤

### Step 1：事件信息收集
- 读取安全事件报告
- 收集：发现时间、泄露途径、数据类型、估计影响范围
- 确认是否已遏制

### Step 2：初步评估
- 评估数据敏感程度
- 判断影响范围
- 确认报告时限

### Step 3：生成监管报告草稿
- 按PIPL要求撰写报告
- 包含：事件描述、原因分析、数据类型和数量、影响评估、已采取措施
- 标注72小时时限节点

### Step 4：生成个人通知草稿
- 撰写给受影响个人的通知
- 说明：泄露信息类型、可能风险、已采取措施、补救建议
- 提供联系方式

### Step 5：创建事件记录
- 建立泄露事件档案
- 记录时间线和所有决策
- 保存以备审计

---

## 输入要求
- 安全事件报告或初步信息
- 涉及的数据类型清单
- 受影响个人范围（如已知）

## 输出
- 监管报告草稿（PIPL格式）
- 个人通知草稿
- 事件记录模板
- 补救建议清单

---

## 注意事项（Pitfalls）
1. **时限错过**：72小时从发现起算，需精确记录发现时间
2. **报告不完整**：遗漏数据类型或影响评估
3. **个人通知延误**：无法通知时应及时公告
4. **证据保存**：事件发现后应立即固化证据
5. **公关风险**：同步考虑对外沟通，避免声誉损失

---

## 验证方式
- 确认报告符合PIPL格式要求
- 核对72小时时限计算
- 验证通知内容完整清晰
