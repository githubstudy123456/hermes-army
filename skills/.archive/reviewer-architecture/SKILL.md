# Reviewer Architecture

## Steps

1. **接收代码交付** — 获取架构设计文档和代码
2. **模块结构审查** — 检查：
   - 目录结构是否清晰（是否按功能/层/模块划分）
   - 模块间耦合度（依赖是否单向，有无循环依赖）
   - 单一职责（每个模块是否只做一件事）
3. **设计模式检查** — 是否合理使用了：
   - 工厂模式、策略模式、观察者模式
   - 依赖注入
   - Repository 模式
4. **数据库设计** — 表结构、索引、外键是否合理
5. **扩展性评估** — 新功能是否需要大量修改现有代码
6. **输出架构审查报告** — 包含设计评分和问题列表

## Pitfalls

- 不要只检查语法错误，要深入逻辑层审核
- 审核报告必须包含具体代码位置，不能只说"有问题"
- 严重问题（P0/P1）必须立即打回，不接受"先上线再改"

## Verification

- 审核报告写入 `~/.hermes/army-workspace/06-审核报告/`
- 报告命名格式：`reviewer-{date}-{skill_name}.md`
- 报告必须包含：审核对象、质量评分、问题列表、修改建议、通过/拒绝结论
