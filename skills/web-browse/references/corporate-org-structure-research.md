# 企业内部组织架构调研方法论

## 核心挑战

搜索"X公司组织架构、部门负责人、P9/P10汇报线"这类内部信息，公开渠道几乎拿不到。字节跳动官方从不公开职级和汇报线。

## 可用信息源排序

| 来源 | 能拿到什么 | 成功率 |
|------|-----------|--------|
| 员工技术博客（博客园、CSDN、知乎） | 产品线、业务模块、团队规模 | 中 |
| 招聘 JD（拉勾、Boss直聘、猎聘） | 岗位名称、职级要求、团队归属 | 高 |
| 行业分析文章（36氪、晚点、极客公园） | 高管变动、业务拆分、组织调整 | 中 |
| 求职社区（脉脉、牛客） | 薪资职级、团队氛围、入职打听 | 低（需登录） |
| 公司财报/招股书 | 业务线划分、员工人数 | 高（但不细化岗位） |
| LinkedIn（高管履历） | 教育背景、任期、职责描述 | 中 |
| 官方招聘页面的 Team 介绍 | 部门分类、大团队分组 | 中 |

**关键发现**：博客园这类技术社区反而有高质量内容（`www.cnblogs.com/GKLBB/p/18927939`），作者以字节产品为框架做系统性整理，信息密度高。

## 推断型架构图的构建方法

当直接信息找不到时，用以下逻辑构建合理推断：

1. **职级体系**：先摸清目标公司的职级对照（如字节 P9=高级总监，P10=资深总监，M4=GM）
2. **业务模块**：从产品矩阵反推组织模块（抖音电商 = 商家+用户+交易+内容+物流+商业化）
3. **行业通用逻辑**：电商平台通用模块（供应链/用户/交易/内容/商业化/技术/支撑）适用于任何电商业务线
4. **汇报线常识**：M4汇报给M5/M6 → P10汇报给M4 → P9汇报给P10，是常见的三级结构

## 已知踩坑

- **内部 org chart 从不公开**：这是公司核心机密，外部无法拿到真实的"谁向谁汇报"
- **财报只披露业务线**：不披露具体岗位和人数
- **脉脉/领英需要登录态**：服务器环境下无法使用
- **百度搜索有滑块验证**：在服务器无头环境下完全无法绕过，换必应

## 适用场景

- 用户要求"对比 X 公司和 Y 公司的组织差异"
