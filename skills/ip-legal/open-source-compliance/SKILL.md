# Open Source Compliance · 开源许可合规

## 基本信息
- **Skill Name**: open-source-compliance
- **Description**: 审计软件开源许可合规性，识别许可证义务与风险
- **Trigger**: 代码引入前审查 / 定期合规审计 / 第三方审计
- **Category**: ip-legal

---

## 核心能力

### 1. 主要开源许可证分类

#### 传染性许可证（Copyleft/LGPL）
| 许可证 | 义务 | 风险 |
|--------|------|------|
| GPL v2/v3 | 修改版本必须开源，使用相同许可证 | 高：商业产品如嵌入可能强制开源 |
| LGPL v2/v3 | 链接库可闭源，修改部分需开源 | 中：动态链接风险较低 |
| AGPL | GPL + 网络使用也需开源 | 高：SaaS服务适用 |

#### 宽松许可证（Permissive）
| 许可证 | 义务 | 风险 |
|--------|------|------|
| MIT | 保留版权声明即可 | 低 |
| BSD 2/3-Clause | 保留版权声明，BSD许可证不得用于宣传 | 低 |
| Apache 2.0 | 保留版权声明，附上许可证和变更说明 | 低 |
| ISC | 与MIT类似 | 低 |

#### 其他常见许可证
| 许可证 | 特点 |
|--------|------|
| MPL 2.0 | 文件级传染，比GPL宽松 |
| Eclipse | 与GPL类似但更宽松 |
| CC0 | 公共领域，无版权 |
| CC BY/CC BY-SA | 创作共用，SA版本有传染性 |

### 2. 合规义务清单
典型许可证合规要求：
1. **版权声明**：保留所有许可证中的版权声明
2. **许可证副本**：分发时应附上许可证文本
3. **修改说明**：修改代码需注明
4. **开源声明**：商业产品需披露使用的开源组件
5. **源代码提供**：Copyleft许可证要求提供源代码

### 3. 审计流程
```
代码扫描 → 组件识别 → 许可证识别 → 义务分析 → 风险评级 → 合规建议
```

## 操作步骤

### Step 1：收集代码信息
- 读取项目SBOM（软件物料清单）
- 或扫描源代码仓库
- 提取第三方组件列表

### Step 2：许可证识别
- 匹配每个组件的许可证类型
- 识别许可证版本
- 检测许可证冲突（同一项目多许可证）

### Step 3：义务与风险分析
- 对照许可证条款分析义务
- 评估商业使用风险
- 识别传染性风险点

### Step 4：生成合规报告
- 输出组件-许可证矩阵
- 风险评级（RED/YELLOW/GREEN）
- 合规建议（替换/保留/申请商业许可）

### Step 5：制定合规方案
- 替代组件建议
- 商业许可证申请指引
- 许可证升级路径（如GPL v2→v3）

---

## 输入要求
- 项目SBOM或代码仓库路径
- 第三方组件清单
- 商业产品信息（用于评估影响）

## 输出
- 开源许可证合规报告
- 风险组件清单
- 合规整改建议

---

## 注意事项（Pitfalls）
1. **许可证版本差异**：GPL v2和v3不兼容，同一代码库混用有问题
2. **动态链接判定**：LGPL通过动态链接可规避开源义务，但需确认链接方式
3. **聚合软件**：独立程序聚合不一定触发传染，但需律师确认
4. **出口管制**：部分开源软件受美国出口管制（如cryptographic），需单独评估
5. **专利授权**：Apache 2.0包含专利授权，需注意

---

## 验证方式
- 核对所有组件均有许可证标识
- 确认风险评级逻辑正确
- 验证合规报告完整性
