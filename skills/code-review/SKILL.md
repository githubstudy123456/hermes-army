# Code Review · 代码审核工作台

## 核心定位

统一的代码审核工作台，覆盖六大审核维度。接收代码路径，按所选维度执行结构化审核，输出标准化报告至 `~/.hermes/army-workspace/06-审核报告/`。  

> 审核报告必须包含：审核对象、质量评分（10分制）、问题列表（附代码位置）、修改建议、通过/拒绝结论。  
> P0/P1 严重问题必须立即打回，不接受"先上线再改"。

## 代码质量审核

### 触发条件
- 主人要求审核代码质量
- 触发词：`代码质量`、`code quality`、`lint`、`命名规范`

### 执行步骤
1. **接收代码路径** — 确认要审核的代码目录或文件路径
2. **静态分析** — 运行 ESLint / Ruff 对代码扫描
   ```bash
   # JavaScript/TypeScript
   npx eslint . --format=json > /tmp/eslint-report.json

   # Python
   ruff check . --output-format=json > /tmp/ruff-report.json
   ```
3. **代码审查** — 逐文件审查，检查以下维度：
   - 命名规范（变量/函数/类名是否有意义）
   - 代码重复（是否有重复代码块 > 3 行）
   - 模块化（函数长度是否 < 50行，模块是否单一职责）
   - 注释质量（关键逻辑是否有注释，注释是否解释 Why 而非 What）
4. **质量评分** — 按 10 分制给出评分：
   - 9-10：优秀，可直接通过
   - 7-8：良好，有小问题但不影响
   - 5-6：一般，需要修改才能通过
   - < 5：较差，需要大幅重构
5. **生成报告** — 填写审核报告模板

### 质量评分标准
| 分数区间 | 风险等级 | 含义 |
|---------|---------|------|
| 9-10 | 🟢 优秀 | 可直接通过 |
| 7-8 | 🟢 良好 | 有小问题但不影响 |
| 5-6 | 🟡 一般 | 需要修改才能通过 |
| <5 | 🔴 较差 | 需要大幅重构 |

## 架构审查

### 触发条件
- 主人要求架构审查
- 触发词：`架构审查`、`architecture`、`模块耦合`、`设计模式`

### 执行步骤
1. **接收代码交付** — 获取架构设计文档和代码
2. **模块结构审查** — 检查：
   - 目录结构是否清晰（是否按功能/层/模块划分）
   - 模块间耦合度（依赖是否单向，有无循环依赖）
   - 单一职责（每个模块是否只做一件事）
3. **设计模式检查** — 是否合理使用了：
   - 工厂模式、策略模式、观察者模式
   - 依赖注入
   - Repository 模式
4. **数据库设计** — 表结构、索引、外键是否合理
5. **扩展性评估** — 新功能是否需要大量修改现有代码
6. **输出架构审查报告** — 包含设计评分和问题列表

## 性能审计

### 触发条件
- 主人要求性能审计
- 触发词：`性能审计`、`performance`、`复杂度`、`慢查询`

### 执行步骤
1. **接收代码** — 确认要审计的性能关键代码
2. **算法复杂度** — 检查是否有 O(n²) 以上或死循环
   ```bash
   # 搜索嵌套循环
   grep -r "for.*for" --include="*.py" --include="*.js" .
   ```
3. **数据库查询** — 检查是否有 N+1 查询、全表扫描
   ```bash
   # 检查慢查询
   EXPLAIN ANALYZE <query>
   ```
4. **资源管理** — 检查：
   - 文件/连接是否正确关闭
   - 是否有内存泄漏
   - 异步任务是否正确 await
5. **Lighthouse 审计** — 对 Web 应用运行性能测试
6. **输出性能报告** — 包含瓶颈列表和优化建议

## 安全漏洞扫描

### 触发条件
- 主人要求安全扫描
- 触发词：`安全扫描`、`security`、`OWASP`、`漏洞`

### 执行步骤
1. **接收代码路径** — 确认要审计的代码
2. **依赖扫描** — 检查 package.json / requirements.txt 中的依赖版本
   ```bash
   # Node.js
   npm audit --json > /tmp/npm-audit.json

   # Python
   pip-audit -r requirements.txt --format=json > /tmp/pip-audit.json
   ```
3. **密钥检查** — 搜索硬编码密钥、API Key、密码
   ```bash
   grep -r "api_key\|password\|secret\|token" --include="*.py" --include="*.js" --include="*.ts" .
   ```
4. **OWASP 检查** — 重点检查：
   - SQL/NoSQL 注入
   - XSS 跨站脚本
   - CSRF 跨站请求伪造
   - 敏感数据暴露
   - XML 外部实体（XXE）
   - 访问控制失效
5. **输出安全报告** — 严重程度分级：P0(立即修)/P1(当天修)/P2(排期修)

## 技术债务评估

### 触发条件
- 主人要求评估技术债务
- 触发词：`技术债务`、`tech debt`、`过期依赖`、`重复代码`

### 执行步骤
1. **代码扫描** — 使用工具识别技术债务
   ```bash
   # 代码重复检测
   ruff check . --select=PLR0912 --output-format=json  # Python

   # 过期依赖检查
   npm outdated
   pip list --outdated
   ```
2. **债务识别** — 检查：
   - 重复代码 > 3 处
   - 过期依赖（1年以上未更新）
   - 硬编码配置
   - TODO/FIXME/HACK 注释
   - 缺失的类型标注（TypeScript/Python type hints）
3. **债务量化** — 按影响程度分级：
   - 高：随时可能引发问题
   - 中：维护成本高
   - 低：可以接受但需要记录
4. **还款计划** — 为每个高/中风险债务制定修复计划
5. **输出债务报告** — 债务清单 + 还款优先级

## 测试覆盖评估

### 触发条件
- 主人要求评估测试覆盖
- 触发词：`测试覆盖`、`coverage`、`单元测试`、`用例补充`

### 执行步骤
1. **接收代码** — 确认要评估的代码项目
2. **覆盖率扫描** — 运行覆盖率工具
   ```bash
   # Node.js
   npx jest --coverage --json > /tmp/coverage.json

   # Python
   pytest --cov=. --cov-report=json > /tmp/coverage.json
   ```
3. **覆盖率分析** — 检查：
   - 行覆盖率（是否 ≥ 60%）
   - 分支覆盖率（关键逻辑是否有分支测试）
   - 函数覆盖率（公开 API 是否都有测试）
4. **测试质量** — 检查：
   - 是否有边界值测试
   - 是否有异常情况测试
   - 测试是否独立（无执行顺序依赖）
5. **补充建议** — 列出缺失的测试用例
6. **输出覆盖报告** — 覆盖率数据 + 补充建议清单

## 通用输出格式

所有维度审核报告写入 `~/.hermes/army-workspace/06-审核报告/`，命名格式：
```

```

```
【质量/评分】X/10（{等级}）

【问题列表】
| 序号 | 严重程度 | 位置 | 问题描述 | 修改建议 |
|------|---------|------|---------|---------|

【修改建议】
1. ...

【结论】
✅ 通过 / ❌ 拒绝（需修改后重新提交）
```

## 通用 Pitfalls

- 不要只检查语法错误，要深入逻辑层审核
- 审核报告必须包含具体代码位置，不能只说"有问题"
- 严重问题（P0/P1）必须立即打回，不接受"先上线再改"
- 审核结论必须明确，不能模糊表述
