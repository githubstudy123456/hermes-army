# Hermès 部门协作工作台

> 每个 Agent 接到任务就负责，做不了也要如实输出报告。不推活，不跳过，不假装理解。

## 核心协作原则

### 1. 任务必须接，做不了如实说
- 上级派任务 → 下级必须接
- 能做 → 做好，输出成果文档
- 做不了 → 输出"升级报告"（说明卡在哪里、需要什么支持），上报

### 2. 文档必须读，下游必须引用上游
- 下一节点必须读上一节点的输出文档
- 在报告里写"根据 [文档名] 第X点..."，不能"根据我的理解"
- 没收到文档就问，不凭猜测执行

### 3. 每个节点输出两类文档之一
```
正常完成  →  [任务名]_[节点ID]_成果.md
无法完成  →  [任务名]_[节点ID]_升级报告.md
```

### 4. 秘书是强制门禁
- 执行层产出必须秘书审核
- 审核通过 → 往上下传
- 不通过 → 打回，写明问题
- 升级报告直达高管，不拦截

## 标准文档模板

### 成果文档
```markdown
# [成果类型] — [任务名]

## 任务来源
- 上游节点：[节点ID]
- 接收时间：[时间]
- 原始任务：[复述任务]

## 成果摘要
[3句话内说明核心产出]

## 详细内容
[按任务要求逐项展开]

## 产出清单
| 产出物 | 路径/位置 | 状态 |
|--------|-----------|------|
| ... | ... | ✅完成 |

## 向下游的说明
[告诉下一个节点需要注意什么]
```

### 升级报告
```markdown
# 升级报告 — [任务名]

## 任务来源
- 上游节点：[节点ID]
- 接收时间：[时间]
- 原始任务：[复述任务]

## 无法完成的原因
[清晰说明卡在哪里]

## 已尝试的解决方式
[做了什么努力]

## 需要什么支持
- 需要资源：[具体资源]
- 需要上级协调：[说明]
- 需要其他部门配合：[部门+事项]

## 当前状态
🔴 阻塞中 / 🟡 需要决策 / ⏳ 等待中

## 建议下一步
[给上级的建议]
```

## 节点流转规则

```
Commander 派发任务
    ↓
高管接收 → 复述确认任务 → 分发给执行层
    ↓
执行层 Agent
    ├── 能做 → 输出成果文档
    └── 做不了 → 输出升级报告 → 汇报高管
    ↓
秘书审核（执行层成果）
    ├── 通过 → 传给高管汇总
    └── 不通过 → 打回执行层重做
    ↓
高管汇总 → 输出本领域汇总报告 → 传给 Commander
    ↓
Commander 整合 → 输出决策备忘录 → 汇报主人
```

## 六部门职责边界

| 部门 | 负责领域 | 典型任务 |
|------|---------|---------|
| COO | 运营/内容/市场/产品/QA | 用户研究、内容生产、市场推广、流程管理 |
| CTO | 技术/架构/开发/安全/运维 | 技术方案、代码开发、系统安全、故障处理 |
| CIO | 数据/AI/研究 | 数据方案、AI模型、算法研发、技术调研 |
| CFO | 财务/投资/合规财务 | 预算测算、投资分析、财务合规、成本控制 |
| CLO | 法律/合规/政策 | 合同审查、监管合规、知识产权、法务风险 |
| CRO | 系统监察/架构健康 | 工作流通畅度、上下文水位、Profile容量、系统架构审计 |

> **CRO 特殊说明**：CRO 不做内容产出，专门负责监察其他部门的运转状态。
> 发现问题直接上报 Commander，不走秘书门禁（因为 CRO 本身就是监察角色）。

## 工作流入口

主人发来任何指令，Commander 先判断走哪个工作流：

| 指令类型 | 工作流文件 | 说明 |
|---------|-----------|------|
| 跨部门复杂任务 | `ceo-org-delegation.yaml` | 最通用模板 |
| 紧急故障/事故 | `incident-response.yaml` | 按类型分流 |
| 产品/功能上线 | `product-launch.yaml` | 5阶段标准流程 |
| 市场推广活动 | `marketing-campaign.yaml` | 策划→执行→复盘 |
| 重大代码变更 | `code-review.yaml` | 多维度评审 |
| **部门监察/专项审计** | `cro-audit.yaml` | **四大监测：工作流/上下文/容量/架构** |

---

## DAG 执行说明

工作流里的 `condition` 字段是**激活条件**，不是"做不做"的选择：
- condition=true → 该节点激活，**必须执行并输出成果或升级报告**
- condition=false → 该节点不激活，跳过

每个节点输出后，下游节点必须读取其文档后才能开始工作。

## CRO 监察体系（四大监测）

### 监测维度

| 维度 | 监控内容 | 判定标准 |
|------|---------|---------|
| **工作流通畅度** | 卡点/堵点/秘书积压/升级截留 | 🟢正常 / 🟡关注 / 🔴卡点P0 |
| **上下文水位** | 各Profile上下文占用/溢出事件 | 🔴>90%溢出 / 🟡70-90%警告 / 🟢<70% |
| **Profile容量** | 容量不足/冗余/覆盖缺口 | 任务积压+成员不足=不足 / >30天无任务=冗余 |
| **系统架构健康** | DAG流转/门禁有效/cron健康/汇报链路 | 🟢优秀 / 🟡合格 / 🔴不及格 |

### 告警分级

| 级别 | 触发条件 | 上报对象 | 时限 |
|------|---------|---------|------|
| **P0** | 上下文溢出 / 工作流彻底卡死 | Commander 立即通知 | 即时 |
| **P1** | 节点超时2x / 上下文>90% / 秘书积压>3 | 对应部门高管 + CRO追踪 | 24h |
| **P2** | 容量不合理 / 协作阻塞 | CRO 记录，周期性汇报 | 每周 |
| **P3** | 架构优化建议 | 纳入季度评审 | 每季度 |

### CRO 工作流结构

```
触发（定时巡检 / 触发告警 / 专项审计）
    ↓
[阶段一] 触发确认（范围/目标/项目清单）
    ↓
[阶段二] 四大监测并行
    ├─ workflow_smootness_check（工作流通畅度）
    ├─ context_water_level_check（上下文水位）
    ├─ profile_capacity_check（Profile容量）
    └─ system_architecture_check（系统架构健康）
    ↓
[阶段三] 秘书汇总初审
    ↓
[阶段四] CRO 终审（审计报告 → 升级/整改/归档）
```
